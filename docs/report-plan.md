# M19: 帳票/印刷 `@banto/report` 計画書

作成日: 2026-07-16  
状態: 実装済み（単位A・単位B完了、2026-07-16）  
提供形態: `@banto/report` パッケージ + 削除可能な最小デモページ
（roadmap.md「M19〜M21 の提供形態」の決定に従う）

## 1. 目的

Markdown ベースの帳票テンプレートにデータをバインドし、印刷 CSS の付いた
HTML として描画・印刷する帳票エンジンを提供する。日報・写真帳・時系列帳票を
「同一エンジンの帳票定義違い」として扱えるようにする（過去アプリ棚卸しで
出現頻度最多の要求。banto-industrial の記録計 R3 が消費予定）。

## 2. スコープ

### 2.1 やること

- 帳票テンプレート言語（Markdown サブセット + データバインド構文）の
  パーサとレンダラ（ヘッドレス・完全テスタブル）
- 印刷用 CSS（A4 縦/横、改ページ制御、表ヘッダーの各ページ繰り返し）と
  画面プレビュー（用紙風の枠）
- `ReportView.svelte`（プレビュー表示 + 印刷ボタン = `window.print()`）
- items データから日報1枚を出す削除可能なデモページ + E2E 1シナリオ

### 2.2 やらないこと（v1 非スコープ）

- ネイティブ PDF 生成（ブラウザ / WebView2 の印刷ダイアログの
  「PDF に保存」で賄う。将来のサーバサイド PDF はバックログ）
- チャート（SVG）の帳票埋め込み（挿入ポイント構文はバックログ。
  v1 の帳票は文字・表・画像で構成する）
- 帳票定義の GUI エディタ、テンプレートの DB 保存・管理画面
- ページ番号・総ページ数の描画（CSS `@page` カウンタのブラウザ差が
  大きい。v1 はヘッダー/フッターの固定文言のみ）

## 3. 設計

### 3.1 テンプレート言語 — Markdown サブセット（自前実装）

**外部 Markdown パーサ（marked 等）は導入しない。** 帳票テンプレートに
必要なのは見出し・段落・表・リスト・強調・画像だけで、CommonMark の全機能
はむしろ有害（HTML 直書きや複雑な入れ子は帳票の再現性を壊す）。制約された
グラマーを自前実装することで、chart/dock と同じフルスクラッチ方針を保ち
つつ「何が書けるか」を仕様として固定できる。

対応ブロック（v1 グラマー、これ以外はテキスト扱い）:

| 構文 | 意味 |
|---|---|
| `# ## ###` | 見出し h1〜h3 |
| 空行区切りの文 | 段落 |
| `- 項目` | 箇条書き |
| `\| a \| b \|` + 区切り行 | 表（2行目 `\|---\|---\|`、右寄せ `---:` 対応） |
| `---` | 水平線 |
| `**太字**` `*斜体*` | 強調（インライン） |
| `![代替](src)` | 画像 |

**生 HTML はエスケープしてテキスト扱い**（テンプレート起因の XSS 面を
構文レベルで閉じる）。

### 3.2 データバインド構文

| 構文 | 意味 |
|---|---|
| `{{ path.to.field }}` | 値の挿入（常に HTML エスケープ） |
| `{{ value \| yen }}` | フォーマッタ適用（既定: `yen` / `number` / `date`。`formatters` オプションで追加登録可） |
| `{{#each path}}` 〜 `{{/each}}` | 配列の繰り返し（行単位のブロック構文。表の行を囲めば行リピート） |
| `{{#if path}}` 〜 `{{/if}}` | 真値のとき出力 |

表の行リピート例（日報の在庫僅少表）:

```markdown
| 商品名 | 在庫 |
|---|---:|
{{#each lowStock}}
| {{ name }} | {{ stock | number }} |
{{/each}}
```

- パスは `.` 区切りのプレーンな参照のみ（式・関数呼び出しは書けない。
  計算はデータを作る側 = アプリのコードで行う）
- 未定義パスは空文字（エラーにしない。ただし `renderReport` の戻り値に
  警告一覧を含め、開発時に気づけるようにする）

### 3.3 パッケージ構成（`packages/report`、生ソース参照構成）

| ファイル | 役割 |
|---|---|
| `src/core/parse.ts` | テンプレート → AST（DOM 非依存・純関数） |
| `src/core/bind.ts` | AST + データ → 解決済み AST（each/if 展開、値解決、フォーマッタ） |
| `src/core/html.ts` | 解決済み AST → HTML 文字列（全テキストをエスケープ） |
| `src/core/index.ts` | `renderReport(template, data, options): { html, warnings }` |
| `src/print.css` | 印刷/プレビュー CSS（§3.4） |
| `src/ReportView.svelte` | プレビュー + 印刷ボタン。`{@html}` は自前エンジンの出力（全値エスケープ済み）のみに使用 |

時刻・乱数を使わない純関数群のため、テストは文字列 in/out の
スナップショット不要の等値比較で書ける。

### 3.4 印刷 CSS と プレビュー

- `@page { size: A4; margin: 12mm }`（`ReportView` の `orientation`
  prop で landscape 切替）
- 帳票面は**テーマ非依存の白地・黒文字固定**（印刷物の再現性優先。
  Light/Dark/Glass の影響を受けない。プレビュー枠の外側だけがテーマに従う）
- 表: `thead` は各ページで繰り返し（ブラウザ標準挙動）、行の途中改ページ
  禁止（`break-inside: avoid`）
- `<hr>` は改ページ（`break-after: page`）として扱う — MD の `---` が
  そのまま「ここで改ページ」の紙感覚で書ける
- 印刷時は帳票以外（シェル・ボタン）を `@media print` で非表示

### 3.5 デモ配線（削除可能）

- items 一覧の PageHeader actions に ghost ボタン「日報」を追加 →
  `(app)/items/report/+page.svelte`（新規1ルート）
- ページ内容: items 全件から集計した日報（日付・総件数・在庫合計・
  カテゴリ別集計表・在庫僅少一覧）を `$lib/banto/reports/daily.md?raw`
  テンプレート + `renderReport` で描画し `ReportView` で表示
- 集計はダッシュボードの既存ヘルパ（`$lib/banto/dashboard`）を再利用し、
  新しい集計ロジックは書かない
- 削除手順（ボタン1行 + ルート1ディレクトリ + テンプレート + package.json
  依存）を README「オプション資産の削除」へ、template-scope §3 に行追加

### 3.6 E2E

スモークに1シナリオ追加: items → 「日報」→ 見出し・集計値・表の行が
描画されていることを確認（`window.print()` は呼ばない）。

## 4. 実装単位

| 順序 | 内容 | 主な対象 | 規模 | 状態 |
|---|---|---|---|---|
| A | ヘッドレスコア（parse / bind / html + テスト） | `packages/report/src/core` | M | 完了 |
| B | ReportView + 印刷CSS + デモページ + E2E + docs | `ReportView.svelte`, `items/report`, README ほか | S–M | 完了 |

実装メモ（単位B）:

- `@page` の A4/landscape 切替は `ReportView.svelte` が所有（print.css は
  `.report-body` のコンテンツ規則のみ）。ベースの `@page { size: A4;
  margin: 12mm }` は常時、landscape 時のみ `<svelte:head>` 経由で
  `@page { size: A4 landscape }` を追加注入し、カスケードで上書きする
  （同一画面に複数の異なる向きの ReportView が同時に印刷される想定は
  v1 スコープ外）。
- 印刷時のシェル（サイドバー/ヘッダー）非表示は、`ReportView` がマウント
  中だけ `document.body` に `banto-report-active` クラスを付与し、
  `apps/admin-template/src/app.css` 側の `@media print` がそのクラス配下の
  `.shell aside` / `.shell header` を隠す方式（`ReportView` 自体はアプリの
  シェル構造を知らない - クラスの存在だけを契約にする）。
- 日報デモの `{{#if lowStock}}` は常に真になる構成（spec §3.5 の想定通り、
  1,000件シード・在庫0〜499の一様分布では在庫僅少が必ず発生する）。

## 5. 検証計画

- コアのユニットテスト: 各ブロック構文、インライン、each/if の入れ子、
  表の行リピート、フォーマッタ、未定義パス警告、**エスケープ
  （データ値に `<script>` を含むケース、テンプレート内の生 HTML）**
- `pnpm check` / `lint` / `test` / `build`、E2E スモーク（11 + 1）、
  visual 40件に回帰なし
- 印刷レイアウトの目視確認は Playwright の `page.pdf()`（headless
  Chromium で可）で A4 出力を生成して確認する（成果物はコミットしない）

## 6. 完了条件

1. 日報デモが画面プレビューされ、印刷ダイアログから A4 で崩れず出力できる
2. 表の行リピート・フォーマッタ・改ページ（`---`）が動作する
3. データ値の HTML・テンプレートの生 HTML が実行されない（XSS テスト）
4. デモを README の手順どおり外すとビルド・テスト・E2E が引き続き通る
5. 既存スイート全てに回帰がない

## 7. 規定値・決定事項（レビュー時の確認ポイント)

| 項目 | 決定 | 根拠 |
|---|---|---|
| Markdown パーサ | 自前の制約付きサブセット（依存追加なし） | §3.1。フルスクラッチ文化と帳票の再現性 |
| バインド構文 | Mustache 風 `{{ }}` + each/if + パイプフォーマッタ | 見た目が非開発者にも読める。式は書けない設計で安全側 |
| PDF | v1 はブラウザ印刷（PDF保存）のみ | WebView2/Chromium で実用十分。ネイティブ生成はバックログ |
| 帳票面の配色 | 白地・黒文字固定（テーマ非依存） | 印刷物の再現性。プレビュー枠のみテーマ追従 |
| デモの入口 | items 一覧の ghost ボタン「日報」 | ナビ項目を増やさず1行で消せる |
