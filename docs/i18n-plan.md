## 0. なぜ今 i18n か — §4.3 の「実例」トリガ

[template-scope §4.3](template-scope.md) は汎用 i18n 辞書層を「2026-07-12 バックログ
から削除確定（**必要になった実例が出たら再考**）」としていた。今回その実例が出た:

> 国内でも外国人労働者が増えており、テンプレートで作った業務アプリを**現場で使う
> エンドユーザー自身が日本語を読めない**ケースが現実的になった（技能実習・特定技能等）。

重要な点: この実例は**アプリ利用者（現場ワーカー）向けのUIランタイム多言語化**を指す。
保守者・採用者向けのドキュメント英語化（AD-6 の docs 部分）とは別レイヤであり、本書の
中心は前者（UI i18n）。§4.3 の当時の判断（「必要な利用者は自前の仕組みを選びたがる」）は、
**現場ワーカーを抱える業務アプリでは i18n が共通要件になる**という今回の実例で覆る。

## 1. レイヤ分割と現在のスコープ（2026-07-22 確定）

i18n は「テンプレ側 vs アプリ側」の二択ではなく、**置き場の異なる3レイヤ + docs** に分かれる。
判断軸: **アプリが自分のコピー内で後付けできるものはアプリ側・実需で保留、共有パッケージ
`@banto/*` に埋まっていてアプリが後付けできないものだけテンプレ側**。

| レイヤ | 中身 | 置き場 | 現在のスコープ |
| --- | --- | --- | --- |
| **①** | `@banto/*` パッケージ内の可視文言 | **テンプレ側（唯一・必須）** | **今やる（本書の実装対象）** |
| **②** | i18n の仕組み（`t()`・ロケール store・provider 配線・言語切替UI） | テンプレ側だが**削除可能** | 保留（設計は本書に温存、実需で着手） |
| **③** | 辞書・対象言語（`en.ts`/`ja.ts`/ワーカーの言語） | **アプリ側** | 各アプリが実需で追加 |
| **docs** | 保守者docs（conventions/recipe/ADR）の英語化 | テンプレ側 | 後回し（他の改修が落ち着いてから。内容が変わり得るため） |

- **なぜ①だけ先にテンプレ側でやるか**: アプリは `@banto/*` を**消費するだけ**で、中の
  ハードコード文言をフォークせずには差し替えられない。将来アプリが多言語化したくなった
  ときに詰むのは①だけで、しかも①は**後付けが全パッケージ横断で高コスト**（＝唯一の後悔
  ポイント）。②③は各アプリが自分のコピー内で後付けできるので保留してよい。
- **①の性質**: 「パッケージが生文言をハードコードせず、**現行の日本語をデフォルトに残した
  まま**注入で上書き可能にする」だけ。②の仕組みにも③の辞書にも依存しない。したがって
  **単一言語（日本語のみ）の採用者には見た目の変化ゼロ・継承負担ほぼゼロ**。
- **初期言語（②着手時）**: 日本語（既存）+ **英語**（基準ロケール）から。①段階では既定は
  日本語のまま（言語選択UIは②）。
- **非対象（当面）**: ②仕組み・③辞書・docs英語化・自動翻訳・RTL・全ロケールの日付/通貨/数値
  フォーマット・3言語目以降。いずれも実需ドリブンで①の上に追加する。

## 2. 不変条件との整合（ここが設計の肝）

i18n は全文言に触れるため、既存の不変条件を壊しやすい。以下を守る:

- **conventions §5（パッケージはアプリ固有 import を持たない）**: `@banto/*` は翻訳辞書や
  `getLocale()` を**import しない**。文言は既存の注入パターン（`AttachmentsPanel` が
  `client` を受け取るのと同型）で、**ラベル props または翻訳関数 `t` を注入**する。
  パッケージは「英語のデフォルト文言 + 上書き可能な props」を持つ形にし、辞書はアプリ層が
  所有する。→ [verify:architecture](../scripts/verify-architecture.mjs) の
  `no-app-import` を壊さない。
- **conventions §10（provider 分岐は provider 層に閉じる）**: ロケール解決・永続化は
  provider/setup 層に閉じ、UIコンポーネントはロケール判定を持たない。トークン文化
  （§9: 生値をコンポーネントに書かない）の**文言版** — 「生の日本語文字列をコンポーネント
  に書かず、キー経由にする」を新しい規約として conventions に1節追加する。
- **conventions §3 / [ADR-0002](adr/0002-minimal-dependencies.md)（依存を足さない）**:
  i18n ランタイムを外部ライブラリにするか自前にするかは §3 の判断基準
  （総保守コスト最小化）に照らす — §4.5 の未決事項。
- **両経路対称（§1）**: i18n はフロント層の関心事で REST/Tauri のサービス層・認可・監査には
  触れない（サービス層は文言を持たない）。監査 detail は機械可読な英語キーのまま
  （§6: detail に秘密を入れない、は維持）。→ 両経路対称の不変条件には影響しない。

## 3. 設計方針

### 3.1 文言のキー化

- アプリ層に辞書を置く（例 `src/lib/banto/i18n/{en,ja}.ts` — フラットな
  `key → string` マップ）。キーは英語ベースの名前空間（`items.list.empty` 等）。
- コンポーネントは `t('items.list.empty')` で引く。`t` は setup 層が構築して
  context/store で配る（既存の provider 注入と同じ経路）。
- 補間は最小限の `{name}` 置換から始める（英語+日本語は複数形の分岐がほぼ不要）。

### 3.2 packages の文言注入（§5 準拠）＝ レイヤ①（今回の実装対象）

**方針決定（2026-07-22）**: ①は「翻訳関数 `t` の注入」ではなく、**現行の日本語をデフォルトに
残した上書き可能な props / メッセージバンドル**で実現する。理由: ①は②の仕組み（`t`・ロケール
store）に**依存させない**のが要件（②は保留中）。`t` 注入は実質②を引き込むため採らない。
既存の `submitLabel = '保存'`（`BantoForm`）や `def.placeholder ?? '選択してください'`
（`SelectField`）が既にこの prop パターンで、①はその横展開。

**粒度**: コンポーネントは可視文言を1つの `messages` オブジェクト props で受け、各キーに
**現行の日本語をデフォルト**として与える（例:
`let { messages = {} }: { messages?: Partial<GridMessages> } = $props();` →
`messages.empty ?? 'データがありません'`）。純関数モジュール（`validate.ts` 等）は
モジュール定数の代わりに**メッセージマップを引数/設定で受け取れる**ようにする。**辞書は
packages に持たせない**（§4 empty-deps・§5 no-app-import を維持）。状態所有権規約
（ロード/空/エラーはコンポーネント内部が所有）とも両立する（文言だけ外部化、状態は内部）。

**棚卸し（2026-07-22、実際の可視文言リテラル。記号 ▸▾▲▼ ≥ ≤ は対象外）**:

| パッケージ / ファイル | 可視文言 | 外部化の形 |
| --- | --- | --- |
| `grid-svelte/FilterPopover.svelte` | 演算子ラベル（含む/で始まる/一致する）・`値を入力`・`${header}の絞り込み` | `messages` props |
| `grid-svelte/BantoGrid.svelte` | `入力値が不正です` | `messages` props |
| `grid-svelte/HeaderCell.svelte` | `${header}の絞り込み` | `messages` props |
| `grid-svelte/core/group.ts` | `(なし)`（空グループ） | 関数引数/設定 |
| `grid-svelte/core/csv.ts` | `${header}: 値を変換できません（${raw}）` | 関数引数/設定 |
| `grid-svelte/core/schema.ts` | 必須/最小/最大/文字数/形式 の各文言 | 関数引数/設定 |
| `forms/validate.ts` | 必須/最小/最大/文字数/形式 の各文言 | 関数引数/設定 |
| `forms/SelectField.svelte` | `選択してください` | prop（既に一部対応） |
| `forms/BantoForm.svelte` | `保存`（submitLabel、既に prop） | 既存 prop |
| `dock-svelte` / `attachments` / `report` / `charts` | 各数個（②着手前に同様に棚卸し） | 同上 |

**注記（重複の解消候補）**: バリデーション文言が `grid-svelte/core/schema.ts` と
`forms/validate.ts` に**重複**している。①の外部化に合わせて共通のメッセージ形へ揃えるか、
片方に寄せるかは実装時に判断（ただしパッケージ間 import は §4/§5 で禁止のため、共通化は
「同じ props 形にする」までで、共有モジュール化はしない）。

**受け入れ条件（①）**: パッケージの可視文言がすべて注入経由になり、デフォルト日本語のまま
`pnpm check` / `cargo test` / `pnpm e2e` / `pnpm e2e:visual` が緑（見た目の回帰なし）。
アプリ側は既定のまま何も渡さなくてよい（後方互換）。

### 3.3 ロケール解決と永続化

- 永続化は既存 `SettingsProvider`（M12）に相乗り（UI設定と同じ経路。localStorage は
  初回描画のちらつき防止 + demo フォールバック、既存 vibrancy/theme と同じ扱い）。
- 3-way provider（tauri/server/demo）すべてでロケール取得を provider 層に閉じる（§10）。
- SSR/初回描画: SvelteKit の静的ビルドで `<html lang>` と初期辞書を早期に確定し、
  テーマと同様にちらつきを抑える。

### 3.4 メッセージフォーマット

- 初期（JA+EN）は単純補間で足りる。**ICU MessageFormat（複数形・性・選択）は、
  複雑な複数形規則を持つ言語（ロシア語・アラビア語等）を追加する実需が出た時点で導入
  検討**（§3 判断基準の「エッジケースが多い領域」に該当し始めたら依存採用も俎上に）。
- 日付/数値: 既存の自前日付変換（§3）はロケール非依存。表示ロケール対応が要る画面が
  出たら `Intl`（ブラウザ標準、依存ゼロ）を使う。

### 3.5 自前実装 vs 依存（未決 → §6）

- §3 の判断基準に照らすと、**JA+EN の単純辞書 + `{name}` 補間なら自前実装が妥当**
  （小さい・エッジケース少ない）。**複数形/性の分岐が必要な言語を足す段で、成熟した
  i18n ライブラリ（例: 型安全な軽量ランタイム）の採用を再評価**する。
- この段階的方針は「依存ゼロが目的ではなく総保守コスト最小化」（§3、P1-5）と一貫。

## 4. マイルストーン

### 4.0 今回の対象（レイヤ①のみ）

1. **packages の可視文言を注入対応化**（§3.2 の棚卸し順、grid → forms → dock/attachments/
   report/charts）。各コンポーネントに `messages` props（現行日本語をデフォルト）、純関数
   モジュールにメッセージマップ引数を追加。**②の仕組み・③辞書は作らない**。
2. **アプリ側の後方互換確認**: 既定のまま何も渡さず、見た目が日本語のまま不変であることを
   `pnpm e2e:visual` で確認（回帰なし＝スナップショット再生成不要）。
3. **（①完了後に判定）機械検査**: 「packages に未注入の生可視文言がない」を
   verify:architecture に追加できるか実現性判定（§9 生色値検査と同型）。

### 4.1 ②以降（保留・実需で着手する将来形）

- **②土台**: `t` 関数 + ロケール store + setup 層配線 + SettingsProvider 永続化 +
  `<html lang>` + デモ経路フォールバック → アプリ層の文言キー化（シェル → items →
  ダッシュボード）→ 言語切替UI（設定画面、visual ベースライン再生成が要る）。
- **③辞書**: 各アプリが `en.ts`/`ja.ts`/ワーカー言語を追加（アプリ側）。
- **docs 英語化**: 「他の改修が落ち着いてから」着手（内容が変わり得るため。2026-07-22 オーナー
  決定）。README 英語版の拡充 → conventions → add-resource recipe → ADR。日本語一次・英語
  追随の同期規律を決める。

## 5. 検証

- `pnpm check`（型: `t` のキー型安全性を可能な範囲で）/ `cargo test`（サービス層は不変、
  回帰なしの確認）/ `pnpm e2e`（言語切替のスモーク1本）/ `pnpm e2e:visual`（切替後の
  ベースライン再生成）。
- 機械検査（§6 の新規検査案）: 「app 層コンポーネントに生の日本語文字列リテラルがない
  （辞書ファイルを除く）」を verify:architecture に追加できれば、§9 の生色値検査と同型の
  「文言はキー経由」規約を機械強制できる。実現性は土台実装後に判定。

## 6. 未決事項

### 6.0 レイヤ①（今回）— 決定済み

- **文言注入の粒度** → **決定: `messages` props（現行日本語デフォルト）+ 純関数はメッセージ
  マップ引数**。`t` 注入は②を引き込むため①では採らない（§3.2）。
- **①段階のデフォルト言語** → **決定: 現行の日本語のまま**（見た目の変化ゼロ）。base-locale
  （英語）の確定は②着手時に持ち越し。

### 6.1 ②以降（保留・②着手前に決める）

1. **i18n ランタイムを自前実装にするか外部ライブラリにするか** — 推奨は「JA+EN は自前、
   複雑な複数形言語の追加時に再評価」（§3.5）。②着手時にオーナー確認。
2. **文言キーの命名規約と粒度** — 名前空間の切り方（画面単位 / リソース単位）。
3. **「番頭 / Banto」ブランド名の翻訳可否** — 製品名は翻訳せず固定する想定（要確認）。
4. **日本語一次 vs 英語一次** — 初期対象は英語だが、辞書のソースオブトゥルースをどちらの
   言語にするか（英語基準ロケールとしたので英語一次が自然だが、既存文言は日本語）。
5. **conventions への新節** — 「UI文言はキー経由・生文言をコンポーネントに書かない」を
   §9（トークン）に並ぶ不変条件として追記する文面（②で文言をキー化した後に確定）。

## 7. 非スコープ（初期は入れない・実需ドリブン）

- 自動機械翻訳の組み込み、翻訳管理SaaS連携、RTL レイアウト、全ロケールの
  日付/通貨/数値フォーマット網羅、3言語目以降の実文言。いずれも土台の上に、実際の
  導入先・言語の実需が出た時点で追加する（本リポジトリの実需ドリブン方針と一貫）。
