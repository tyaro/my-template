## 0. なぜ今 i18n か — §4.3 の「実例」トリガ

[template-scope §4.3](template-scope.md) は汎用 i18n 辞書層を「2026-07-12 バックログ
から削除確定（**必要になった実例が出たら再考**）」としていた。今回その実例が出た:

> 国内でも外国人労働者が増えており、テンプレートで作った業務アプリを**現場で使う
> エンドユーザー自身が日本語を読めない**ケースが現実的になった（技能実習・特定技能等）。

重要な点: この実例は**アプリ利用者（現場ワーカー）向けのUIランタイム多言語化**を指す。
保守者・採用者向けのドキュメント英語化（AD-6 の docs 部分）とは別レイヤであり、本書の
中心は前者（UI i18n）。§4.3 の当時の判断（「必要な利用者は自前の仕組みを選びたがる」）は、
**現場ワーカーを抱える業務アプリでは i18n が共通要件になる**という今回の実例で覆る。

## 1. スコープ

- **対象**: テンプレートアプリ層（`apps/admin-template/src/`）と、UI文言を持つ
  `packages/@banto/*` コンポーネント（grid の空状態、forms のバリデーション文言、
  dashboard パネル名等）のユーザー可視文言。
- **初期言語**: 日本語（既存）+ **英語**の2言語から始める。英語を「基準ロケール」に据える。
- **docs 英語化**（AD-6 の別レイヤ）: conventions / add-resource recipe / ADR 等の
  保守者docsを英訳。UI i18n とは独立に進められるため本書では概要のみ、詳細は
  adoption-plan AD-6 で管理。
- **非対象（初期）**: 自動翻訳連携、RTL（アラビア語等）、全ロケールの日付/通貨/数値
  フォーマット網羅、3言語目以降の実文言。いずれも土台の上に**実需ドリブンで追加**する。

## 2. 不変条件との整合（ここが設計の肝）

i18n は全文言に触れるため、既存の不変条件を壊しやすい。以下を守る:

- **conventions §5（パッケージはアプリ固有 import を持たない）**: `@banto/*` は翻訳辞書や
  `getLocale()` を**import しない**。文言は既存の注入パターン（`AttachmentsPanel` が
  `client` を受け取るのと同型）で、**ラベル props または翻訳関数 `t` を注入**する。
  パッケージは「英語のデフォルト文言 + 上書き可能な props」を持つ形にし、辞書はアプリ層が
  所有する。→ [verify:architecture](../scripts/verify-architecture.mjs) の
  `no-app-import` を壊さない。
- **conventions §10（provider 分岐は provider 層に閉じる）**: ロケール解決・永続化は
  provider/setup 層に閉じ、UIコンポーネントはロケール判定を持たない。トークン文化
  （§9: 生値をコンポーネントに書かない）の**文言版** — 「生の日本語文字列をコンポーネント
  に書かず、キー経由にする」を新しい規約として conventions に1節追加する。
- **conventions §3 / [ADR-0002](adr/0002-minimal-dependencies.md)（依存を足さない）**:
  i18n ランタイムを外部ライブラリにするか自前にするかは §3 の判断基準
  （総保守コスト最小化）に照らす — §4.5 の未決事項。
- **両経路対称（§1）**: i18n はフロント層の関心事で REST/Tauri のサービス層・認可・監査には
  触れない（サービス層は文言を持たない）。監査 detail は機械可読な英語キーのまま
  （§6: detail に秘密を入れない、は維持）。→ 両経路対称の不変条件には影響しない。

## 3. 設計方針

### 3.1 文言のキー化

- アプリ層に辞書を置く（例 `src/lib/banto/i18n/{en,ja}.ts` — フラットな
  `key → string` マップ）。キーは英語ベースの名前空間（`items.list.empty` 等）。
- コンポーネントは `t('items.list.empty')` で引く。`t` は setup 層が構築して
  context/store で配る（既存の provider 注入と同じ経路）。
- 補間は最小限の `{name}` 置換から始める（英語+日本語は複数形の分岐がほぼ不要）。

### 3.2 packages の文言注入（§5 準拠）

- `@banto/grid-svelte` / `@banto/forms` 等は**英語をハードコードのデフォルト**にし、
  各文言を props で上書き可能にする（例: `<BantoGrid emptyLabel={t('...')} />`）。
- あるいは軽量な「翻訳関数 `t?: (key) => string` を1つ注入」する形。**辞書そのものは
  packages に持たせない**（§4 empty-deps・§5 no-app-import を維持）。
- どちらの粒度（個別props vs `t` 注入）にするかは §5 未決事項。既存の状態所有権規約
  （ロード/空/エラーはコンポーネント内部が所有）と両立させる。

### 3.3 ロケール解決と永続化

- 永続化は既存 `SettingsProvider`（M12）に相乗り（UI設定と同じ経路。localStorage は
  初回描画のちらつき防止 + demo フォールバック、既存 vibrancy/theme と同じ扱い）。
- 3-way provider（tauri/server/demo）すべてでロケール取得を provider 層に閉じる（§10）。
- SSR/初回描画: SvelteKit の静的ビルドで `<html lang>` と初期辞書を早期に確定し、
  テーマと同様にちらつきを抑える。

### 3.4 メッセージフォーマット

- 初期（JA+EN）は単純補間で足りる。**ICU MessageFormat（複数形・性・選択）は、
  複雑な複数形規則を持つ言語（ロシア語・アラビア語等）を追加する実需が出た時点で導入
  検討**（§3 判断基準の「エッジケースが多い領域」に該当し始めたら依存採用も俎上に）。
- 日付/数値: 既存の自前日付変換（§3）はロケール非依存。表示ロケール対応が要る画面が
  出たら `Intl`（ブラウザ標準、依存ゼロ）を使う。

### 3.5 自前実装 vs 依存（未決 → §6）

- §3 の判断基準に照らすと、**JA+EN の単純辞書 + `{name}` 補間なら自前実装が妥当**
  （小さい・エッジケース少ない）。**複数形/性の分岐が必要な言語を足す段で、成熟した
  i18n ライブラリ（例: 型安全な軽量ランタイム）の採用を再評価**する。
- この段階的方針は「依存ゼロが目的ではなく総保守コスト最小化」（§3、P1-5）と一貫。

## 4. マイルストーン（実装着手時の順序）

1. **土台**: `t` 関数 + ロケール store + `en`/`ja` 辞書スケルトン + setup 層配線 +
   SettingsProvider 永続化 + `<html lang>`。デモ経路のフォールバック。
2. **アプリ層の文言キー化**: シェル（サイドバー/ヘッダー/ログイン/設定）→ items デモ →
   ダッシュボード。英語文言を基準に、日本語を対訳で用意。
3. **packages の文言注入化**: grid/forms/その他の可視文言を props/`t` 注入に移行。
   verify:architecture に「packages に生文言をハードコードしない/辞書を持たない」検査を
   追加できるか検討（§6）。
4. **言語切替UI**: 設定画面にロケール選択（theme プリセット選択と同じ場所・同じ provider
   経路）。visual regression のベースライン再生成が要る（設定画面はスナップショット対象）。
5. **docs 英語化（AD-6 別レイヤ、並行可）**: README 英語版の拡充（現 `README.en.md` は
   要約1枚）→ conventions → add-resource recipe → ADR。日本語一次・英語追随の同期規律を
   決める（UI i18n の辞書ドリフトとは別管理）。

## 5. 検証

- `pnpm check`（型: `t` のキー型安全性を可能な範囲で）/ `cargo test`（サービス層は不変、
  回帰なしの確認）/ `pnpm e2e`（言語切替のスモーク1本）/ `pnpm e2e:visual`（切替後の
  ベースライン再生成）。
- 機械検査（§6 の新規検査案）: 「app 層コンポーネントに生の日本語文字列リテラルがない
  （辞書ファイルを除く）」を verify:architecture に追加できれば、§9 の生色値検査と同型の
  「文言はキー経由」規約を機械強制できる。実現性は土台実装後に判定。

## 6. 未決事項（実装着手前に決める）

1. **i18n ランタイムを自前実装にするか外部ライブラリにするか** — 推奨は「JA+EN は自前、
   複雑な複数形言語の追加時に再評価」（§3.5）。要オーナー確認。
2. **packages の文言注入の粒度** — 個別ラベル props か、`t` 関数1つの注入か（§3.2）。
3. **文言キーの命名規約と粒度** — 名前空間の切り方（画面単位 / リソース単位）。
4. **「番頭 / Banto」ブランド名の翻訳可否** — 製品名は翻訳せず固定する想定（要確認）。
5. **日本語一次 vs 英語一次** — 初期対象は英語だが、辞書のソースオブトゥルースをどちらの
   言語にするか（英語基準ロケールとしたので英語一次が自然だが、既存文言は日本語）。
6. **conventions への新節** — 「UI文言はキー経由・生文言をコンポーネントに書かない」を
   §9（トークン）に並ぶ不変条件として追記する文面。

## 7. 非スコープ（初期は入れない・実需ドリブン）

- 自動機械翻訳の組み込み、翻訳管理SaaS連携、RTL レイアウト、全ロケールの
  日付/通貨/数値フォーマット網羅、3言語目以降の実文言。いずれも土台の上に、実際の
  導入先・言語の実需が出た時点で追加する（本リポジトリの実需ドリブン方針と一貫）。
