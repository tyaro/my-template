## 1. 目的

Banto の機能、画面構成、データフローを維持したまま、管理画面テンプレートとしての
第一印象と日常的な使いやすさを引き上げる。

今回の変更はフロントエンドに限定する。Rust、API、DB スキーマ、認証・権限判定、
データ取得処理は変更しない。Svelte コンポーネントのマークアップ、CSS 変数、
スタイル、アイコン、および表示専用の UI 状態が主な変更対象となる。

目指すのは、見栄えだけを優先したショーケースではなく、データ量の多い業務画面でも
視認性と操作性を保つ「Modern Operations Console」である。

## 2. 現状認識

現在の UI は機能が揃っており、Light/Dark、Standard/Glass、グリッド、フォーム、
チャート、ドッキングといった基盤も CSS 変数を通じて連携している。一方で、外観を
テンプレートの強みにするには次の改善余地がある。

1. `@banto/theme` のトークンが色、単一の角丸、基本寸法を中心としており、面の階層、
   影、状態色、コントロール寸法、モーションを画面ごとに補っている。
2. ボタン、入力欄、カード、補足文のスタイルが複数ページに重複し、画面間で細部が
   揃いにくい。
3. サイドバーとヘッダーは機能的だが、ブランド、現在地、検索、ユーザー操作の視覚的な
   優先順位が弱い。
4. ナビゲーションに絵文字を使っているため、OS によって見た目とサイズが変わる
   （`navigation.ts` 自身が絵文字を「アイコンセット決定までのプレースホルダー」と
   注記している）。
5. ダッシュボードの先頭に長い技術説明があり、KPI やチャートより先に目へ入る。
6. 主要画面にレスポンシブ規則がほぼなく、LAN ブラウザや狭いウィンドウでツールバーが
   窮屈になりやすい。
7. 読み込み中、空状態、エラー、危険操作などの状態表現が画面ごとに異なる。
8. `app.css` が Tailwind v4 を import し依存にも含まれるが、コンポーネント側で
   ユーティリティクラスは実質使われておらず、トークン体系との関係が未定義のまま
   になっている。

## 3. デザイン方針

### 3.1 ビジュアルコンセプト

- ベースは墨色・ネイビーとニュートラルなグレーで構成する。
- 主要操作は既存のブルーを整理したエレクトリックブルーで示す。
- Banto の提灯を想起させる暖色は、ブランドマークや小さな注目点に限定する。
- 角丸、細い境界線、面の明度差、控えめな影で情報の階層を作る。
- 大きな装飾画像には依存せず、タイポグラフィ、余白、アイコン、グラデーションで
  完成度を上げる。
- Glass は標準 UI の代替テーマとして扱い、Glass を選ばないと成立しないデザインには
  しない。

### 3.2 操作密度

Banto は業務用管理画面のテンプレートであるため、過度に大きなカードや余白は採用しない。
標準のコントロール高は 36〜40px を目安とし、KPI、チャート、設定画面では余白を広め、
グリッドとツールバーでは情報密度を維持する。

これに加えて、密度をテーマ軸として持つ（§3.5）。固定値ではなく
`--banto-control-height` 系トークンを密度別に定義する。

### 3.3 テーマ互換性

以下の4組み合わせを同等にサポートする。

| 明暗 | プリセット |
|---|---|
| Light | Standard |
| Dark | Standard |
| Light | Glass |
| Dark | Glass |

Glass の透明面はシェルとカードに限定し、既存方針どおり仮想グリッド内部は不透明に保つ。

### 3.4 Tailwind v4 の扱い（決定）

**保持し、`@theme` でトークンを橋渡しする。**

- `packages/theme` の `--banto-*` トークンを、`app.css` の `@theme` ブロックで
  Tailwind のテーマ変数（`--color-*`、`--radius-*`、`--shadow-*` など）へ
  マッピングする。これにより、テンプレート利用者はユーティリティクラス経由でも
  Banto テーマ（Light/Dark/Glass/密度）に自動追従するスタイルを書ける。
- 本リフレッシュのコンポーネント実装自体は、引き続き scoped `<style>` +
  `--banto-*` 変数で書く（パッケージ側は Tailwind に依存しない、という既存構造を
  維持する）。ユーティリティはアプリ側ページの微調整用と位置づける。
- Tailwind を不要とする利用者向けに、README のテンプレート改変手順へ
  「Tailwind の外し方」（`app.css` の import と `@tailwindcss/vite` の削除）を
  1 項目追加する。

この決定により「トークンは banto.css、余白は Tailwind、影は各ページ」という
三重管理を防ぐ。単一の情報源は常に `--banto-*` トークンである。

### 3.5 密度軸（新規）

Glass プリセット（M12）と同じ「純粋な変数スワップ」方式で、`<html>` に
`data-banto-density="standard" | "compact"` を追加する。

- standard（既定）: コントロール高 36px、グリッド行高 36px（現行値）。
- compact: コントロール高 32px、グリッド行高 32px、余白トークンを 1 段階縮小。
- 対象は寸法トークンのみ。コンポーネントは既に `--banto-grid-row-height` 等を
  参照しているため、追加実装は設定画面のトグルと変数定義に限られる。
- 保存は既存のテーマ設定と同じ経路（localStorage / SettingsProvider）とする。

グリッドの仮想化は行高変数を既に前提としているため計算ロジックの変更は不要だが、
密度切替時に保存済みドックレイアウトやスクロール位置が壊れないことを
確認項目に含める（§7.2）。

## 5. 実装フェーズ

### Phase 0: 表示基準の固定（ツール化する）

基準スクリーンショットは手作業ではなく、**Playwright の
`expect(page).toHaveScreenshot()` によるビジュアルリグレッションプロジェクト**
として整備する。既存 E2E（M18 スモーク、`banto-serve` 実サーバ相手）とは別の
Playwright project とし、決定的なシードデータで撮影する。

- 対象画面: ログイン、ダッシュボード、商品一覧、ユーザー管理、設定、
  コマンドパレット
- マトリクス: Light/Dark × Standard/Glass × 3 ビューポート
  （1440×900、1024×768、768×1024）
- 上記で最大 72 枚となるため、全組み合わせを撮るのはログインとダッシュボードのみ
  とし、他画面は Light/Standard + Dark/Glass の対角 2 組み合わせに間引いてよい
  （それでも規則性の破れは検出できる）。
- アニメーションは `animations: 'disabled'` で固定し、チャートのツールチップ等の
  ホバー状態は対象外とする。
- ベースラインはリポジトリにコミットし、以後の各 Phase の PR で差分レビューに使う。
  意図した変更はベースライン更新をコミットに含める。

狭い幅ではグリッドの横スクロールを許容するが、ページ全体の意図しない横スクロール
は発生させない。

### Phase 1: テーマトークンと共通表現

`@banto/theme` に、既存トークンと互換性を保ったまま **付録 A のトークン仕様表**
のとおり追加する。種類は以下（具体値・コントラスト比は付録 A で確定済み）。

- `surface-subtle`、`surface-hover`、`surface-overlay` など面の階層
- 弱い境界線と強い境界線（強い境界線は対面比 3:1 以上）
- `sm`、`md`、`lg` の角丸
- `sm`、`md`、`lg` の影
- 状態色を「文字/アイコン用」「塗り用」「淡色地+濃色文字のペア」に分離した
  主要、控えめ、危険、成功、警告のセット
- 入力欄とボタンの標準高さ（密度軸 §3.5 に対応）
- 見出し、本文、補助文、数値表示のタイポグラフィ（付録 A.5）
- モーショントークン: `--banto-duration-fast/base/slow` と
  `--banto-ease-out` / `--banto-ease-spring`（CSS `linear()` によるバネ曲線。
  サイドバー開閉・モーダル・トーストに使用し、JS アニメーションライブラリは
  導入しない）

既存の `--banto-radius` などは直ちに削除せず、新トークンへのエイリアスとして残す。
テンプレート利用側の既存オーバーライドを壊さないためである。

現行値のうち **light の `--banto-success: #16a34a` は白文字で 3.3:1 となり
塗りボタンに使えない**ため、付録 A のとおり塗り用は `#15803d` を採用する
（チャート用スロット色は据え置き）。

#### タイポグラフィの先進要素

ネットワークフォント禁止の制約下でも「デザインされた文字」の印象を作る:

- 数値列・KPI: `font-variant-numeric: tabular-nums`
- 日本語見出し・本文: `font-feature-settings: 'palt'`（約物詰め）
- 見出し: `text-wrap: balance`、補足文: `text-wrap: pretty`
- KPI の大数値は `@property` 登録したカスタムプロパティの補間による
  カウントアップ表示（CSS のみ、`prefers-reduced-motion` で無効化）

#### アイコンライブラリ（決定）

**`@lucide/svelte`（Lucide、ISC ライセンス）** を採用する。

- 選定理由: Svelte 5 対応、named export 単位のツリーシェイクが効く、
  アウトラインスタイルが §3.1 の方針と一致、ISC はテンプレート再配布と両立。
- 採用は 1 パッケージのみとし、ビルド出力で未使用アイコンが同梱されていないことを
  確認する（§7.1）。
- 万一ビルド検証で問題が出た場合の代替は、使用アイコンのみを自前 Svelte
  コンポーネント化して `packages/theme` に同梱する方式とする（数十個で足りる）。

#### 共通コンポーネント

アプリ側では、頻出する `PageHeader`、`SurfaceCard`、`StatusBadge`、`IconButton` のうち、
マークアップと振る舞いが共通するものだけを小さな UI コンポーネントとして切り出す。
単なる CSS の再利用だけを目的とした過剰なラッパーは作らない。

これに加えて、**メニュー一式**（`Menu` / `MenuItem` / `MenuSeparator` / `MenuGroup`）を
共通コンポーネントとして自前実装する。フォーカス管理、矢印キー移動、`Escape` での
クローズとフォーカス返却、`aria-haspopup`/`role="menu"` といった振る舞いは画面ごとに
書くと必ずばらつくため、集約する価値が最も高い部類である。

- 表示層はネイティブ **Popover API**（top-layer + light dismiss）を使い、
  外部のヘッドレス UI ライブラリは導入しない（チャート・ドック・コマンドパレットと
  同じ自前主義を維持する）。
- 単一階層のドロップダウンのみ。ネスト式サブメニュー、メニューバー、ホバー展開は
  作らない（必要になった時点で拡張する）。
- 最初の利用者は Phase 2 のヘッダーユーザーメニュー。Phase 4 の狭幅ツールバーの
  オーバーフローメニュー（「…」）も同じ土台に乗せる。
- 将来 `grid-svelte`（行コンテキストメニュー）や `dock-svelte` が共用する昇格候補
  第一号のため、**アプリ固有の import（sessionStore 等）を混ぜずに書く**ことを
  レビュー条件とする。

#### 状態表現の所有権境界

計画が問題視する「状態表現の重複」を再発させないため、所有権を先に決める:

- **パッケージ側**（`grid-svelte`、`forms`、`charts`、`dock-svelte`）:
  自身の内部状態（グリッドの空/読み込み/セルエラー、フォームのフィールドエラー、
  チャートのデータなし）はパッケージが持ち、`--banto-*` 状態色トークンのみを参照する。
- **アプリ側**（admin-template）: ページレベルの状態（一覧 0 件の案内、権限不足、
  ページ全体のエラー、トースト）は共通コンポーネントとして 1 箇所に置き、
  各ページはそれを使う。
- パッケージがアプリの状態コンポーネントに依存する方向の参照は作らない。

### Phase 2: アプリシェル

サイドバー、ヘッダー、メイン領域を一体として刷新する。

#### サイドバー

- 絵文字を Lucide のアウトラインアイコンへ置き換える。
- ブランド領域、主要ナビゲーション、管理系ナビゲーション、ユーザー領域を視覚的に
  分ける。
- アクティブ項目は背景色だけでなく、アクセントラインまたはアイコン色でも示し、
  `aria-current="page"` を付与する。
- 折りたたみ時はアイコン位置を変えず、ラベルをツールチップで補う。
- 幅の切り替え時に本文が不自然に跳ねないよう、幅とラベルの表示を同期させる。

`navigation.ts` は文字列の絵文字ではなく、意味のあるアイコンキーを保持する形に変更し、
実際のアイコンコンポーネントへの対応は表示層で行う。

#### ヘッダー

- 現在のページ名を主見出しとして明確に表示する。
- コマンドパレット入口を検索ピルとして表示し、`Ctrl+K` / `Cmd+K` を常時発見できる
  ようにする。
- ログアウトを裸のテキストボタンからユーザーメニューへ移す。Phase 1 の共通
  `Menu` コンポーネントを使用する（`menu`/`menuitem` ロール、矢印キー移動、
  `Escape` で閉じてトリガーへフォーカス返却）。
- `sessionStore.identity` と `role` の既存情報だけを使い、追加 API は呼ばない。
- 小さい画面では検索ピルをアイコンへ縮小する。

#### メイン領域

- ページごとの余白を共通化する。
- 一覧系は横幅を最大限使い、設定・フォーム系は読みやすい最大幅を設定する。
- 900px 前後を境にサイドバーをオーバーレイ化できる構造へ変更する。

ドックホストは自身のサイズを実測して保存レイアウトを clamp するため、シェル幅の
挙動変更後に**保存済みドックレイアウトの復元**が壊れていないことを本 Phase の
確認項目とする（全体 QA まで先送りしない）。

### Phase 3: ダッシュボード

ダッシュボードを、テンプレートの機能と外観を最初に伝えるショーケースとして整える。

- 長い技術説明は「このデモについて」の折りたたみ可能な補足へ移す。
- 先頭に短いページ説明と最終更新などの軽いメタ情報を配置する。
- KPI カードへ統一アイコン、数値、補足、状態色を追加する。
- 実データから意味のある前期間比較を算出できない項目には、見せかけの増減値を追加しない。
- チャートカードを「タイトル、補足、チャート、凡例・操作」の構造に統一する。
- 重要なチャートを広く、補助チャートを小さくする非対称グリッドを採用する。
- チャート色はテーマトークンから取得し、Light/Dark のコントラストを調整する。
  categorical パレットは「スロット固定・再割当禁止」の既存規約があるため、
  変更する場合はスロット単位の before/after 対応表を PR に含める。
- ドッキング領域は「分析ワークスペース」として通常カード群から視覚的に分離する。
- ドックのタイトルバー、タブ、ドラッグゴースト、スナップ表示を新トークンに合わせる。

データ集計、パネル ID、保存済みドックレイアウトの形式は変更しない。

### Phase 4: 一覧、グリッド、フォーム

#### 商品一覧と監査ログ

- ページタイトルと主要操作を共通 `PageHeader` に揃える。
- 表示モード、グループ化、インポート、エクスポートを、優先度に応じて Primary、
  Secondary、Ghost に整理する。
- 狭い幅では操作群を折り返し、主要操作を先頭に維持する。
- CSV インポート結果は通常カードではなく、成功・警告・失敗が判別できる状態パネルにする。
- 監査ログ詳細は選択行との関係が分かるレイアウトにする。

#### `@banto/grid-svelte`

- ヘッダー背景、境界線、列リサイズ、ソート、フィルター状態を明確化する。
- 行ホバー、範囲選択、アクティブセル、編集中、エラー状態を同じ状態色体系へ揃える。
- 数値列で tabular numbers を使用する。
- 空状態と読み込みプレースホルダーを改善する（所有権は Phase 1 の境界どおり
  パッケージ側）。
- 行高と仮想化計算は変更せず、Glass 内でも不透明面を維持する。

#### `@banto/forms`

- ラベル、必須表示、説明、エラーの垂直リズムを統一する。
- input、select、textarea、checkbox のフォーカスと無効状態を揃える。
- 送信領域を共通化し、処理中の状態が視覚的に分かるようにする。
- 検証と送信ロジックは変更しない。

### Phase 5: 設定、ユーザー管理、ログイン

#### 設定

- 単一の細長いカラムを、画面幅に応じたカテゴリカードのグリッドへ変更する。
- 各セクションにアイコン、短い説明、操作領域を設ける。
- ラジオボタン風のテーマ選択は、選択結果が分かるプレビューカードへ変更する。
- 密度（standard/compact、§3.5）の切替を追加する。
- boolean 設定は共通のスイッチ表現へ揃える。
- 認証無効化、リストア、パスワードなど影響の大きい操作を Danger zone として分離する。

#### ユーザー管理

- 作成、一覧、選択ユーザー編集の関係を明確にする。
- 広い画面では一覧と編集を2カラム、狭い画面では縦積みにする。
- ロールは色だけに依存しないバッジで表示する。
- 削除とパスワードリセットを通常の保存操作から分離する。

#### ログイン

- デスクトップ幅ではブランド領域とフォームの2ペイン構成にする。
- 狭い画面では1カラムのカードへ戻す。
- 背景は画像アセットなしで生成する: 複数の `radial-gradient` の重ね +
  `conic-gradient` によるメッシュ + インライン SVG `feTurbulence`（data URI）の
  微細ノイズ。提灯を想起させる暖色の光彩はここに限定して使う（§3.1）。
- **ブランドペインは差し替え可能な最小構造にする。** README の「テンプレートから
  自分のアプリを作る」手順はログイン画面の Banto 表記差し替えを前提としているため、
  ロゴ・アプリ名・キャッチコピーの 3 スロット程度に留め、作り込んだ装飾を
  ブランド固有にしない。
- デモ認証情報はフォームの説明文から補足パネルへ移す。
- 初回セットアップと通常ログインで同じ外枠を使う。
- 認証処理と初回セットアップ判定は変更しない。

### Phase 6: 状態、モーション、レスポンシブ

- 読み込み、空、成功、警告、エラー、権限不足の表示パターンを統一する
  （所有権は Phase 1 の境界どおり）。
- Toast と Command Palette を新しい面、影、フォーカス表現へ合わせる。
  Command Palette はフォーカストラップと `Escape` の挙動を確認項目に含める。
- ホバー、展開、サイドバー開閉、モーダル表示に控えめな動きを追加する。
  イージングは Phase 1 のモーショントークン（`--banto-ease-spring` 含む）を使う。
- **View Transitions によるページ遷移**を導入する。SvelteKit の `onNavigate` +
  `document.startViewTransition` で、ページ切替時に PageHeader と主要カードが
  クロスフェードする。非対応ブラウザ（LAN 側の古い環境）では従来どおり瞬時切替に
  フォールバックし、JS 分岐は `startViewTransition` の存在チェックのみとする。
- `prefers-reduced-motion: reduce` では View Transitions を含む非本質的な
  アニメーションを無効にする。
- 既存の `prefers-reduced-transparency` による Glass のフォールバックを維持する。
- タッチ操作でも押せるサイズを確保し、ホバーだけに依存する操作を作らない。

## 6. 実装単位

レビューと切り戻しを容易にするため、実装時は次の単位に分ける。規模は相対見積り
（S: 1〜2日、M: 3〜5日、L: 1〜2週間の目安）。

| 順序 | 内容 | 主な対象 | 規模 |
|---|---|---|---|
| 0 | ビジュアルリグレッション基盤 | `e2e/`（screenshot project + ベースライン） | S |
| 1 | テーマ基盤と共通 UI | `packages/theme`, `app.css`（`@theme` 橋渡し含む）, 共通コンポーネント | M |
| 2 | シェルとログイン | Sidebar, Header, layout, login | M |
| 3 | ダッシュボードとチャート/ドック外観 | dashboard, charts, dock-svelte | M |
| 4 | 一覧、グリッド、フォーム | items, audit-log, grid-svelte, forms | L |
| 5 | 設定とユーザー管理（密度切替含む） | settings, users | M |
| 6 | レスポンシブ、状態、View Transitions、全テーマ QA | 全画面 | M |

各単位で機能変更とビジュアル変更を混在させない。特にグリッドとドックは、見た目の変更で
イベント処理や保存形式へ影響を与えないことをレビュー条件とする。

## 7. 検証計画

### 7.1 自動確認

- `pnpm check`
- `pnpm lint`
- `pnpm test`
- `pnpm build`
- 既存 E2E スモーク（10 シナリオ。回帰範囲は限定的である点に留意）
- Phase 0 のビジュアルリグレッション（各 PR で差分レビュー）
- E2E に **axe-core**（`@axe-core/playwright`）による主要画面のアクセシビリティ
  スキャンを追加し、コントラスト・ロール・ラベルの自動検出を行う

アイコン追加後は、未使用アイコンがまとめてバンドルされていないこともビルド出力で確認する。

### 7.2 表示確認

| 観点 | 確認内容 |
|---|---|
| テーマ | Light/Dark × Standard/Glass の4組み合わせ |
| 密度 | standard/compact でグリッド・フォーム・ドック復元が崩れない |
| 幅 | 1440×900、1024×768、768×1024 |
| シェル | サイドバー展開/折りたたみ/オーバーレイ、ドックレイアウト復元 |
| 権限 | admin/editor/viewer でナビ項目と操作が崩れない |
| データ | 通常、0件、読み込み中、エラー、大量行 |
| 操作 | キーボード、フォーカス、Ctrl/Cmd+K、ドラッグ、スクロール、ユーザーメニュー |

主要画面は変更前後を同じデータ、同じビューポートで比較する（Phase 0 の
スクリーンショット基盤を使う）。目視確認では、単独画面の美しさだけでなく、
ページを移動したときにボタン、見出し、カード、余白の規則が一貫していることを重視する。

## 8. 完了条件

以下をすべて満たした時点でビジュアルリフレッシュ完了とする。

1. サイドバー、ヘッダー、ログイン、ダッシュボードから OS 依存の絵文字 UI がなく、
   統一アイコンで表示される。
2. ボタン、入力欄、カード、バッジ、状態表示が同じテーマトークンを利用している。
3. 全テーマ組み合わせで、本文テキストは対面 4.5:1 以上、大きな見出し・アイコン・
   入力欄境界線は 3:1 以上のコントラストを満たす（付録 A の計算値を基準とし、
   axe-core スキャンで違反ゼロ）。
4. 768px 幅で主要操作が欠落せず、ページ全体に不要な横スクロールが発生しない。
5. グリッドの仮想スクロール、セル編集、ドック操作、テーマ・密度保存、認証フローに
   回帰がない。
6. `prefers-reduced-motion` と `prefers-reduced-transparency` が機能する。
7. `check`、lint、test、build、既存 E2E、ビジュアルリグレッション、axe スキャンが
   成功する。
8. 新しいバックエンド API、DB 変更、ネットワークアセットを必要としない。

## 9. 優先順位

短期間で効果を出す場合は、Phase 0〜3 を最優先とする。テーマ基盤、シェル、
ダッシュボードまで完了すれば、起動直後の印象とテンプレートのデモ価値は大きく改善する。

Phase 4〜6 は、そのデザインを日常操作する全画面へ展開し、テンプレートとしての一貫性と
品質を完成させる工程と位置づける。

## 11. v2 候補（本計画では実施しない）

先進的だが本計画のスコープを超えるもの。テンプレートの差別化候補として記録する。

1. **コマンドファースト UI**: Ctrl+K パレット（M16）を「ナビの補助」から
   「第一の操作系」へ昇格。グリッド行・設定項目・最近の操作を横断検索し、
   インラインプレビューを表示する。バックエンド変更なしで到達可能。
2. **アクセント・ヒューのユーザー調整**: `--banto-hue` 1 変数から OKLCH 相対色
   構文で primary 系を導出し、利用者が 1 行でブランド色に染め替えられるようにする。
   非スコープの「テーマエディタ」を作らずにその価値の大半を先取りできるが、
   状態色マトリクス全体の再検証が必要なため v2 とする。
3. **アンビエントステータス**: サイドバーの提灯ブランドマークが SSE 接続中の
   LAN クライアント数やエラー発生を光彩で淡く表現する。SCADA 志向
   （roadmap §5）と相性が良い。
4. **グリッドの IDE 化**: 仮想スクロールバーへの注釈（エラー行・フィルタ一致の
   ヒート表示）、グループヘッダーの sticky 表示、セル内スパークライン。
5. **スクロール駆動の凝縮ヘッダー**: CSS Scroll-Driven Animations でページヘッダー
   が大見出し→コンパクトバーへ連続変形。LAN 側ブラウザの対応差があるため
   フォールバック設計込みで v2。
