## 3. 各項目の詳細

### AD-1: 「対象読者 / 非対象」ポジショニング宣言（新規・最優先）

採用レビュー §4.1 の最大の障壁は「二形態ニッチが汎用テンプレとして誤って受け取られる」
こと。ニッチは**隠すより宣言した方が合う人に速く届く**。README 冒頭（クイックスタートの
直前）に短い節を置く:

- **Banto が向く人**: デスクトップアプリと LAN ブラウザ配信の**両方**が要る業務系
  （現場端末 + 事務所ブラウザ）、認証・RBAC・監査ログ付きの管理画面を最初から欲しい人、
  AI併走で量産したい人。
- **Banto が向かない人**: Webのみ / デスクトップのみで足りる人、React/Electron の人材・
  エコシステムに乗りたい人、大規模DB（Postgres/分散）が前提の人、多言語UIが要件の人。
- **v1の割り切り（正直な開示）**: LANは標準HTTP（TLSはリバプロ終端、[ADR-0003](adr/0003-tls-via-reverse-proxy.md)）、
  DBはSQLiteのみ（Postgresはfeatureスタブ）、UIとdocsは日本語前提。

**受け入れ条件**: 初見の検討者がREADMEの最初の1画面で「自分向きか」を判断できる。
既存の「5分で動かす」（P1-7）はこの節の直後に残す。

### AD-2: 視覚訴求（新規・最優先）

UIテンプレートの採用第一要因は「見た目」。現状READMEにUI画像が一切ない。素材は
既にある — `e2e/visual/visual.spec.ts-snapshots/` に light/dark × standard/glass の
ダッシュボード・items・login・settings・command-palette のスナップショットが揃って
いる。

- **最小案（規模S）**: ダッシュボード（dark-glass と light-standard の2枚）と
  データグリッド（items）のスクリーンショットを README 冒頭に貼る。画像は
  `docs/assets/` 等に配置し、視覚回帰スナップショットとは別管理（スナップショットは
  テスト資産であり、リブランド時に旧見た目で撮り直される前提のため直接埋め込まない）。
- **発展案（規模M）**: LAN配信（QRでスマホ/別端末から同じ画面）を見せる短尺GIF、
  または GitHub Pages 等に**ブラウザ単体デモ**（`pnpm build` の静的出力 = InMemory
  プロバイダで動く）をホストして README からリンク。デモは backend 不要で `pnpm dev` が
  そのまま動く構造（`getBantoMode()` の demo 経路）なので、静的ホスティングと相性が良い。

**受け入れ条件**: README を開いた検討者が、スクロールせずに Banto の見た目を把握できる。
発展案を採る場合、ライブデモURLが README 冒頭にある。

### AD-3: リリースタグ運用の起動 + 信頼シグナル（新規）

現状 git タグ 0 件。[publishing.md](publishing.md) の「タグ運用規約」と
[CHANGELOG.md](../CHANGELOG.md) の `[Unreleased]` 運用は整備済みだが**一度も起動して
いない**。採用検討者にとって「タグ付きリリースがある」は基本的な信頼シグナル。

1. **`v0.1.0` を実際に打つ**: `[Unreleased]` の内容を `[0.1.0] - 2026-07-xx` 節へ
   切り出し、タグを作成（publishing.md の 0.x 規約に従う）。以後 PR ごとの
   `[Unreleased]` 追記と、区切りでのタグ発行を回す。
2. **`SECURITY.md`**（脆弱性報告の連絡先と、LAN HTTP の既知の割り切りへのリンク）。
   セキュリティ意識の高さ（conventions §6）が既にあるので、報告経路を明示するだけで
   シグナルになる。
3. **CONTRIBUTING の方針1枚**（任意）: 「外部PRを受けるか / ハードフォーク前提か」を
   明示。受けない方針なら「Use this template で自分のリポジトリを作る」導線を書く。
   これはトラックB（README）と重複しない運用宣言。

**受け入れ条件**: GitHub の Releases に1件以上のタグ、`SECURITY.md` が存在し README から
辿れる。

### AD-4: コピー面積削減の着手判定（再評価）

[template-scope §7](template-scope.md) は `admin-template-core` の汎用ロジック
（users/audit/settings/backup 約4千行）を `banto-admin-services` クレートへ寄せる方針を
**記録済み・未実施**とし、着手条件を「2本目の利用アプリが立ち上がり実際に重複コピーが
発生した時点、または banto-industrial が相当物を必要とした時点」としている。

採用レビューの追加論点: これは**採用者が最初に払うコスト（継承する保守責任の行数）**
であり、「2本目のアプリ」を待つ条件は**作者内部の都合**で、外部採用者の初日には効か
ない。ただし本リポジトリの「実需ドリブン」文化（[maintainability-review §4.1](maintainability-review-2026-07.md)）を
尊重すると、**まだ着手しない**判断も合理的（誤ったクレート境界を早く切ると保守負担に
なる）。

**本書の推奨**: §7 の着手条件に「**外部採用者から『継承コードが多い』というフィード
バックが観測されたとき**」を第3のトリガとして追記するに留める（今すぐの抽出はしない）。
AD-2 の発展案（デモ公開）で採用者が増えれば、このフィードバックが実際に得られる。
**受け入れ条件**: template-scope §7 の着手トリガに採用者フィードバックが追記される。

### AD-5: リソース追加の儀式削減の着手判定（再評価）

[scaffold-presets-plan.md](scaffold-presets-plan.md) は minimal/standard/full プリセット
生成の**設計のみ完了**で、実装は P2-1 v2（対話式スキャフォールドCLI）の土台待ち。
リソース追加13ファイルの儀式（採用レビュー §3.3）は、この土台があれば大きく縮む。

採用レビューの追加論点: 儀式は**量産フェーズの律速**であり、採用者が「1つ目のリソース
を足す」初日の体験に直結する。ただし P4-9 は規模 L で、AD-1〜AD-3（規模 S〜M で採用
初速に即効）より費用対効果が劣る。

**本書の推奨**: AD-1〜AD-3 を先行し、**AD-2 のデモ公開後に採用者数の反応を見てから**
P2-1 v2 / P4-9 の実装優先度を再評価する。現時点では `add-resource.md` レシピ（AI に
そのまま渡せる）が儀式の実質的な緩和策として機能している点を評価し、CLI 実装は急がない。
**受け入れ条件**: 本書の実施順序（§4）に P4-9 の再評価トリガが記録される。

### AD-6: 深部ドキュメントの英語化 or JP特化の明確化（要判断）

現状: コードコメント・識別子は英語、`README.en.md`（要約1枚）は完了（P2-2）。一方
conventions / recipe / ADR / scope などの**深部ドキュメントは日本語のみ**。これは
「グローバル採用の障壁」であると同時に、「JP特化という戦略の表れ」でもある。**どちらの
戦略を採るかで、やるべき作業が正反対になる**ため、実装前に方針を決める必要がある。

- **選択肢A: グローバル志向** — conventions + add-resource recipe + ADR を1件ずつ英訳。
  母数の大きい英語圏に間口を開くが、二重管理の保守コストが発生し、日本語一次・英語追随の
  ドリフト管理が要る（i18n を UI で退けた [§4.3](template-scope.md) と同じ緊張）。
- **選択肢B: JP特化を明確化** — 深部docsは日本語のままとし、AD-1 のポジショニングで
  「日本語圏の業務開発者向け」と宣言。保守コストを増やさず、合う層に集中する。番頭という
  命名・industrial-plan の存在とも整合的。

**本書の推奨: 選択肢B（JP特化の明確化）を第一候補**とする。理由: (1) i18n非導入
（§4.3）・二形態ニッチ（採用レビュー §4.1）と戦略的に一貫、(2) 単独メンテナで二重管理の
ドリフト管理は割に合わない、(3) 深部docsを読むのはハードフォークして保守を引き継ぐ
少数で、その層は英訳がなくても評価できる。**選択肢Aは「英語圏からの採用実需が実際に
観測されたとき」に再検討**（実需ドリブン）。

**この判断は作者（プロジェクトオーナー）の戦略判断であり、本計画では確定しない。** AD-1
のポジショニング文言がこの判断に依存するため、**AD-1 着手前に決定する**。
**受け入れ条件**: 選択が template-scope または本書に記録され、AD-1 の文言に反映される。

## 4. 実施順序の推奨

1. **フェーズ0（戦略判断、コード変更なし）**: AD-6 の A/B 決定。これが AD-1 の文言を
   決める。所要は議論のみ。
2. **フェーズ1（採用導線のクイックウィン、合計1〜2日）**: AD-1 → AD-2（最小案）→ AD-3。
   いずれも規模 S〜M で、コアに触れず採用の初速に即効。**採用レビューの実質的な処方箋は
   このフェーズに集中する。**
3. **フェーズ2（反応を見て判断）**: AD-2 発展案（ライブデモ公開）→ その後の採用者反応を
   もって AD-4（コピー面積）/ AD-5（scaffold）の着手を実需ドリブンで再評価。
4. **フェーズ3（実需が出たら）**: AD-4 / AD-5 の実装。これらは既存計画（template-scope §7 /
   scaffold-presets-plan / P2-1 v2）の実装そのものであり、本書は着手トリガのみを更新する。

各 PR は roadmap §7 の実施プロセス（CIゲート必須）と template-scope §6 のチェックリストに
従う。conventions の不変条件（特に「依存を足さない」）を破らないこと。AD-2 の画像は
リポジトリ肥大を避けるため最適化した PNG を数枚に留め、ライブデモは backend 不要の
demo 経路（既存の InMemory プロバイダ）を使い新しいランタイム依存を持ち込まない。

## 5. スコープ外 / 変えないこと

採用レビューの過程で確認し、**現状維持と判断**したもの（改善項目に混ぜないための記録）:

- **二形態（デスクトップ + LAN）ニッチそのもの** — 弱点ではなく戦略。汎用化して間口を
  広げるのではなく、AD-1 で**ニッチを宣言して合う層に届ける**方向を採る。
- **アプリUIの日本語ハードコード / i18n非導入**（[template-scope §4.3](template-scope.md)）—
  維持。AD-6 の推奨（JP特化）とも整合。
- **npm/crates.io へ公開しない**（[publishing.md](publishing.md)）— 維持。AD-3 のタグ運用は
  git tag 参照配布の枠内で、レジストリ公開とは別。
- **自前実装文化 / 依存を足さない**（conventions §3、[ADR-0002](adr/0002-minimal-dependencies.md)）—
  維持。AD-2 のデモも新規ランタイム依存を持ち込まない。
- **runtime プラグイン機構を作らない**（[template-scope §3.1](template-scope.md)）— 維持。
  AD-5（scaffold）は削除の自動化であり動的プラグイン化ではない。

本書の打ち止め条件: AD-1〜AD-3 が採用の初速に効く本丸であり、AD-4〜AD-6 は実需/戦略
判断ドリブン。「英語圏や複数採用者からの実需が実際に観測されたとき」以外に AD-4/5/6 の
優先度を機械的に上げない（既存レビューの実需ドリブン方針と一貫）。
