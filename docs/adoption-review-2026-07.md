## 1. 前提 — なぜ「外向き」の角度で見るか

既存レビューは「コードとして良いか／AIが安全に保守できるか」を突き詰めており、
その結論は一貫して高評価だった（Claude ★5多数、ChatGPT 8.3/10）。本書はそこを
繰り返さず、**「技術的に良い＝採用される」ではない**という前提に立つ。テンプレート
の価値は「良さが採用検討者に伝わり、導入の初速が出るか」で決まる。したがって本書の
評価軸は、コードの正しさではなく**導入体験・情報設計・信頼シグナル・ターゲット
適合**である。

結論を先に置く: **エンジニアリングの完成度は公開テンプレートとして上位数%。
一方で「他ユーザーが採用する」魅力は、技術的優秀さとは別の要因（言語・ニッチさ・
実績シグナル・視覚訴求）で頭打ちになっている。** 一言でいえば「作りは一流、間口は
狭い」。

## 3. 観点別評価

### 3.1 AIコーディングから使う — ★★★★★（最大の武器）

AIに委譲する前提の設計が、他のどのリポジトリより徹底している。

**効いている点:**
- `AGENTS.md` の「タスク別の入り口」索引でコンテキスト投入を最小化できる。
- `recipes/add-resource.md` が「そのままプロンプトに使える」チェックリスト形式
  （レシピ自身が「AIに委譲するときは本レシピをそのまま指示に使う」と明記）。
- `verify:architecture` rule 8 が「mutating を片方の経路だけに足す」典型ミスをCIで
  必ず落とす。AIの盲点を機械で塞いでいる。
- Definition of Done 様式が完了報告を定型化し、レビュー入力に直結。

**穴（正直に）:**
- **`src-tauri` がサンドボックス/CIでコンパイルできない**ため、AIがTauriコマンド層
  （両経路対称の片側）を変更しても自分では検証を閉じられない。rule 8 は「両経路に
  存在するか」までで、「認可レベル・監査が実際に同一か」は見ない — AIが最も間違え
  やすく自動検証が届かない領域。※ 週次+パス条件付きの `tauri-check.yml`
  （improvement-plan P3-2）でマージ前検知は一部担保済み。

### 3.2 作者自身が使う — ★★★★☆

- 意思決定が ADR / conventions / コードコメントの3分類で残り、将来の自分が「なぜ」を
  失わない。半年後の自分に最も優しい構造。
- 247+809テストがリファクタの安全網。
- **難点は「コピー面積」＝儀式のコスト。** リソース1つ追加で Rust（migration/service/
  rest/tauri）+ フロント（resource/page/nav/dashboard/sample）の約13ファイルを触る。
  動的ルートを意図的に捨てた判断（「読み解けない魔法を増やさない」）は妥当だが、量産
  フェーズでは律速になる。作者自身も [template-scope §7](template-scope.md) で
  「汎用ロジック約4千行はコピー継承する負債」と認識済み・**未実施**。

### 3.3 他ユーザーが使う — ★★★☆☆

- 「すべては削除可能」+ トラックA（docs）はハードフォーク時に丸ごと削除可、という
  逃げ道設計は良心的。各オプション（dock/charts/glass/palette/attachments/report）に
  削除手順が明文化されている。ここまで面倒を見るテンプレートは稀。
- ただし採用者が最初に払うコスト: (a) コピー面積（汎用RBAC/監査/バックアップ約4千行を
  自分の保守責任下に継承）、(b) リソース追加のたび13ファイル（`scaffold-presets-plan.md`
  は設計のみ・未実装）、(c) 深部ドキュメント（conventions/recipe/ADR）が日本語のみ
  （`README.en.md` は100行要約のみ）。

### 3.4 まとめ（観点1〜3）

内部品質は文句なし。使う側の摩擦は「量（コピー面積・儀式）」と「言語」に集約される。
いずれも既存計画に部分的に項目があるが、**採用の初速という観点では未着手のものが
律速**になっている（→ 計画書 AD-4/AD-5/AD-6）。

## 4. 採用魅力の評価 — ★★★☆☆（技術は★5、到達可能な市場が狭い）

### 4.1 採用の障壁（影響度順）

1. **二形態（デスクトップ + LANブラウザ配信）というニッチさ。** 唯一の差別化点で
   あると同時に間口を絞る。大半の開発者は「デスクトップ」か「Web」の**どちらか**が
   欲しい。両方を1コードベースで欲しいのは**現場端末（キオスク/倉庫/工場）+ 事務所
   ブラウザ**という特定業務。命名（番頭）・[industrial-plan.md](industrial-plan.md)
   の存在からも狙いは業務/現場系。**これは弱点ではなく戦略**だが、「汎用テンプレ」と
   して打ち出すと刺さらない。
2. **言語。** アプリUIは日本語完結（i18n は [template-scope §4.3](template-scope.md)
   で非導入確定）、深部ドキュメントも日本語。事実上ターゲットは日本語話者の業務開発者。
   グローバル採用は現状ほぼ無理。
3. **信頼シグナルの不足。** v0.1.0、単独メンテナ、**タグ付きリリース0件**、
   コミュニティ実績が薄い。配布も npm publish ではなく git tag + `path:` 依存という
   やや特殊な方式（[publishing.md](publishing.md)）。採用検討者は「バスファクター1」
   「自前実装の grid/charts/forms のバグは誰が直すか」を気にする。
4. **技術スタックの人材プール。** Svelte 5 runes + Tauri v2 + Rust は React/Electron
   より母数が小さい。自前UI群はエコシステムの助けを借りにくい。
5. **v1の割り切り。** LANは平文HTTP（TLSはリバプロ前提、[ADR-0003](adr/0003-tls-via-reverse-proxy.md)
   で決定済み）、SQLiteのみ（Postgresはfeatureスタブ）。「業務用」を名乗る割にこの2点は
   採用判断で引っかかりやすい（ドキュメントで正直に開示している点は良い）。
6. **視覚訴求の不在。** UIテンプレートの採用第一要因は「見た目」だが、README冒頭に
   スクリーンショット/デモがない。視覚回帰スナップショット（`e2e/visual/**`）という
   素材はあるのに活かせていない。

### 4.2 魅力として効いているもの

- 認証/RBAC/監査/バックアップ/CSV/コマンドパレットが**デモスタブでなく実装済み**で
  最初から乗っている。ゼロから業務管理画面を作る人には数週間の短縮になる。ここは強い訴求。
- 「二形態が必要」という具体ニーズを持つ層に対しては、市場に競合がほぼいない。

## 5. 総括

「番頭」は、**日本語圏の業務系（現場+事務所ハイブリッド）開発者が、認証・監査付き
管理画面をAI併走で量産する**という具体シナリオに対しては、競合不在レベルの完成度。
現状は「技術的に優れているのに、その優秀さが伝わる導線（デモ・言語・ターゲット宣言・
実績シグナル）が未整備」なため、**採用は技術力ではなくマーケティングと間口設計で損を
している**状態。

したがって次の投資は新機能ではなく、**(A) 採用者に価値を伝える導線（ポジショニング
宣言・視覚訴求・リリース実績）と、(B) 導入摩擦の低減（コピー面積・儀式・言語）**に
向けるべき。既存の内部品質レビューの結論（「次段階は導入体験・検証ギャップ・機械化に
投資」）とも方向が一致する。実施計画は [adoption-plan-2026-07.md](adoption-plan-2026-07.md)。
