# ADR-0002: 依存を足さず自前実装を既定にする（総保守コスト最小化）

- 状態: Accepted
- 日付: 2026-07-19（判断自体は M0 以降の運用で確立、遡及記録）
- 関連: [conventions.md §3](../conventions.md)、
  improvement-plan-2026-07.md P1-5

## コンテキスト

Banto は**コピーして使うテンプレート**であり、依存はすべて利用者が継承・
監査・更新する対象になる。ワークスペースに何を入れる／入れないかは、
バイナリサイズ・監査面・利用者の更新負荷に直結する基盤方針で、機能追加の
たびに判断が要る。

## 決定

**「引きたくなった依存」を既定で入れず、自前実装で代替する。** 現に入れて
いない代表例と代替（conventions §3 の表）: `chrono`/`time` → 手書き日付変換、
MIME 検出ライブラリ → マジックバイト判定、`multipart` → 生バイト body +
`?fileName=`、`tower-http` → 手書き middleware、markdown ライブラリ →
自前パーサ + エスケープ、`tracing` → `eprintln!`。

ただし目的は「依存ゼロ」ではなく**総保守コストの最小化**であり、採否は
P1-5 の基準（自前実装が肥大化する / セキュリティ境界に関わる /
Unicode・日時・暗号・パーサ等エッジケースが多い / crate が十分成熟 /
feature 限定できる / バイナリ増加を測定済み、の複数該当）で判断する。

## 検討した代替案

- **案A（採用）: 自前実装を既定、依存採用は基準付きの例外。** 利点:
  バイナリ肥大・監査面・利用者の更新負荷を抑える／コード全体を把握できる。
  欠点: 日付・MIME・パーサ等エッジケースの多い領域を自前で抱え、
  セキュリティ修正も自力で追う保守負担。
- **案B（不採用）: 必要なら普通に crate を引く（一般的な方針）。** 実装は
  速いが、テンプレート利用者全員がその依存ツリーを継承・監査することになり、
  「コピーして使う」前提と噛み合わない。小さな utility のために依存ツリーが
  膨らむのを常態化させるため却下。
- **案C（検討したが不採用の一般化）: 依存を一律禁止。** 硬直的すぎる。
  Unicode 正規化や暗号など、自前実装がかえって高リスク・高コストになる
  領域まで禁止すると総保守コストが上がる。よって「一律禁止」ではなく
  「基準付きの抑制」（案A）にした。

## 帰結

- 上表のいずれかを引きたくなったら、それは設計判断であり**議論の対象**。
  P1-5 の基準に照らして是非を判断し、採用するなら理由を残す
  （conventions §3）。
- 既存の自前実装を予防的に置き換えることはしない。各実装が基準に実際に
  該当し始めた時点（肥大化・仕様拡張・脆弱性追従が重くなった等）で個別に
  再評価する。
- pre-commit フックに `husky`/`lint-staged` を入れず素の POSIX sh に
  したこと（README「pre-commit フック」）も、本 ADR の具体的な適用例。
