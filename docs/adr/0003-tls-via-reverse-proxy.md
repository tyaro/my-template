# ADR-0003: LAN の TLS はリバースプロキシ終端を正式サポートとし、組み込み TLS は保留する

- 状態: Accepted
- 日付: 2026-07-19
- 関連: [conventions.md §3](../conventions.md)、[ADR-0002](0002-minimal-dependencies.md)、
  spec §11.2、improvements.md §2.3、improvement-plan-2026-07.md P1-4/B、
  README「LANアクセス」

## コンテキスト

組み込み Web サーバ（`banto-server`）は v1 では**平文 HTTP + トークン認証**
という割り切りで、ログイン情報・セッショントークンが暗号化されずに LAN を
流れる（spec §11.2）。信頼できる LAN 以外での運用や拠点をまたぐ利用には
TLS が要る。どう TLS を提供するかは、依存最小化（ADR-0002）とセキュリティの
トレードオフを伴う基盤判断になった。

## 決定

**TLS は Banto 本体に組み込まず、リバースプロキシ（Caddy 等）での終端を
正式サポート経路とする。** Banto 自身は HTTP のまま `127.0.0.1` バインドに
絞り、前段のプロキシが TLS を終端する。README に Caddy の構成例と、
プロキシ経由で per-IP レート制限が縮退する注意を記載済み（P1-4）。
組み込み TLS（rustls オプトイン）は**保留**とし、下記の再検討条件を満たした
時点で改めて判断する。

## 検討した代替案

- **案A（採用）: リバースプロキシ終端。** 利点: 新規依存ゼロ（ADR-0002 と
  整合）、TLS 設定・証明書更新・OCSP 等の複雑さを実績あるプロキシに委ねる、
  自動 HTTPS（Let's Encrypt / 内部 CA）が Caddy 数行で済む。欠点: 運用者が
  もう1プロセス動かす必要があり、デスクトップ利用者向けのゼロコンフィグ
  ではない。
- **案B（保留）: 組み込み rustls オプトイン + 自己署名証明書自動生成
  （rcgen 等）。** 利点: ゼロコンフィグでデスクトップに優しい。欠点:
  (1) rustls + 証明書生成という**重い依存追加**（ADR-0002 の抑制対象、
  バイナリ肥大・監査面拡大）、(2) 自己署名証明書は各ブラウザで警告が出て、
  利用者が CA を手動インストールする**信頼 UX の問題**を Banto が抱え込む、
  (3) TLS 設定 UI・証明書ローテーションという保守面の増設。ゼロコンフィグ
  の魅力に対しコストが大きく、案A が本番 TLS を既に満たすため現時点では
  見送る。
- **案C（不採用）: 平文のみで TLS 経路を用意しない。** 完全に信頼できる
  LAN 以外では通用せず、実運用者に逃げ道がない。無責任なので却下（案A で
  逃げ道を用意する）。

## 帰結

- README「LANアクセス」は Caddy によるリバースプロキシ TLS を手順として
  持つ。バインドは `127.0.0.1` に絞る前提（`0.0.0.0` だとプロキシを迂回した
  平文 HTTP でも届く）。
- 組み込み TLS を実装するなら、それは独立したマイルストーンとして、
  Tauri をビルド・TLS ハンドシェイクを実地検証できる環境で行う（本テンプレの
  サンドボックスでは src-tauri をコンパイルできず未検証出荷になるため）。
- **再検討条件**: 「ゼロコンフィグのデスクトップ TLS」への強い需要があり、かつ
  自己署名証明書の信頼 UX（CA 配布 or 内部 CA 前提）を Banto 側で引き受ける
  合意ができたとき。その際は rustls を feature ゲートし、最小ビルドは
  依存ゼロを保つ（ADR-0002 の一般則）。
- spec §11.2 の「v1 は平文 HTTP」という割り切りは本 ADR で追認された状態。
