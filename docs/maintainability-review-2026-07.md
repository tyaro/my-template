## 1. 前提 — 人間の保守性とAIの保守性の分岐点

本プロジェクトの保守・機能追加は AI コーディング中心で進む。大半の保守性
（小さい関数・明確な境界・テスト・隠れたグローバル状態を持たない）は人間と
共通だが、縁の部分で分岐する。本レビューはこの分岐を踏まえて重み付けした。

1. **記憶の持ち方が逆。** 人間は数ヶ月かけて頭にモデルを蓄積し「grid-core は
   存在しない」と*知っている*。AIは毎回コールドスタートで**今読めるものから**
   モデルを再構築する。→ ドキュメントドリフトは人間よりAIに危険（古い記述を
   信じる）。**散文の規約を「失敗する検査」に変換するほどAI保守性が上がる。**
2. **AIは字義的でドキュメントを信頼する。** 機械強制されていない散文の不変
   条件は、AIにとって「守られている保証がない」。**機械強制 > 散文規約**の
   原則が人間以上に効く。
3. **フィードバックループがAIの生命線。** AIは `src-tauri` をサンドボックスで
   コンパイルできない。**決定的・密閉・高速な検査（unit / verify-architecture /
   型）だけがAIの目**。起動しないと検証できない領域（Tauri経路・両経路対称）は
   AIの盲点で、テストや明示契約で余分に守る価値がある。
4. **パターン密度・明示性 > 賢い抽象。** AIは周辺コードにパターンマッチして
   拡張する。均一で反復的でgrep可能なコードは、多少の重複を含んでもAIに易しい。
   Banto の「依存を足さない・書き下す」文化と両経路での監査重複は、実は強く
   AI保守的であり、DRY化より**現状維持が正解**。
5. **「なぜ」だけは再導出できない。** AIはコードから「何を」は速く読めるが、
   書かれなかった「なぜ」（退けた代替案・順序制約）は復元できない。→ load-bearing
   コメントとADRは高価値。

## 2. 総評

Rustサービス+サーバ層は**すでに強くAI保守的**。`crates/banto-storage/src/list_query.rs`
のモジュールdocは模範例で、マクロ採用の理由（sqlxのトレイト境界と戦うコスト）・
「未知sortは無視/未知filterはhard error」という意図的な非対称・`ColumnMap`
ホワイトリストのセキュリティモデルまで、再導出できない「なぜ」がすべて書かれて
いる。この水準を全体で守れば良い、という状態。

したがって本計画の指摘は「悪いコード」ではなく、**AIが変更すると壊れるのに
自動で検出されない不変条件**（conventions.md のうち機械検査に落ちていないもの）に
集中する。

### 保護すべき「モデル」（消さない判断）

- **`list_query.rs` の "why" doc** — 他の自前実装（`backup.rs` のステージング適用、
  `banto-server/src/auth.rs` の2次元スロットル）も同水準の意図コメントがあるか、
  という基準で評価する。
- **両経路での監査呼び出しの重複** — サービス層に寄せてDRYにする誘惑があるが、
  AI前提では各経路に明示的に書いてある方が安全（片方だけ変えた diff が見える）。
  **現状維持。**
- **1概念1ファイルの命名**（`rest/items.rs`・`itemsAdmin.ts`）— P3-1/P3-4 の分割は
  AIポジティブ。継続。

## 3. 中心的所見 — 不変条件の機械検査カバレッジのギャップ

conventions.md の不変条件のうち `scripts/verify-architecture.mjs` が守っているのは
約半分（rule 1〜6 + rule 7 ドキュメント整合性）。残りは散文・レビュー・手書き
テストだけが頼りで、そこがAIの盲点。

| § | 不変条件 | 機械検査 | AIが壊すリスク | 機械化の余地 |
| --- | --- | :---: | :---: | --- |
| **1** | **REST/Tauri 両経路対称** | ✗ | **最高** | 中：両経路の mutating コマンド/ルート集合の1:1対応 + 各々の監査呼び出しを照合 |
| 2 | サービス層非依存 | △(tauri/axum のみ) | 中 | 小：RBAC/HTTP型の混入も同手法で追える |
| 3 | 依存を足さない | △(packages の空deps のみ) | 中 | 中：crate/app の依存追加も差分検出可 |
| **6** | **セキュリティ横断** | ✗(一部テスト) | **高** | 部分的：「`NewAttachment` に mime 無し」「`settings_get`/`settings_set` のロール対称」等は grep 可 |
| 8 | Svelte 5 runes 落とし穴 | ✗ | 中 | 小〜中：既知パターンの一部 lint 可 |
| 10 | 3-way provider + demo 明示拒否 | ✗ | 中 | 中：provider 分岐に demo の明示 throw があるか検査可 |
| 11 | マイグレーションの流儀 | ✗ | 低 | 小：連番・不可逆の規約チェック |
| 12 | doc は仕様節を参照 | ✗ | 低 | 小 |

**最重要は §1（両経路対称）。** このテンプレートの背骨の不変条件でありながら、
AIが「片方の経路にだけ mutating コマンドを足す」ミスをしても**落ちる検査が
一つもない**。しかもAIは `src-tauri` をコンパイルできず、Tauri側の破綻を実行でも
気づけない。今は 2307行の `rest/tests.rs` と人間のペア実装が唯一の砦。

## 4. 改修計画（優先順）

いずれも「散文の規約 → 落ちる検査」への変換で、rule 7（docs 整合性）と同じ方向。
`verify-architecture.mjs` に追加し、依存を足さない（Node 標準のみ、conventions §3）。

| # | 項目 | 対象 § | 規模 | 状態 |
| --- | --- | --- | --- | --- |
| **CR-1** | **両経路対称の機械検査**（rule 8）— Tauri コマンドの mutating 集合と REST ルートの mutating 集合を突き合わせ、対応が崩れたら CI で落とす | §1 | S〜M | **完了（2026-07-21）** |
| CR-2 | §6 の grep 可能なセキュリティ不変条件を検査化（rule 9）— `NewAttachment` に mime 無し / `settings_get`・`settings_set` のロール対称 等、明確に機械化できるものだけ | §6 | S | **完了（2026-07-21）** |
| CR-3 | §3 の依存追加検出を crate/app にも拡張 — `empty-deps` の思想を Cargo/app 側の許可リスト方式に広げる | §3 | M | **見送り（実需ドリブン）** — §4.1 |
| CR-4 | §2 の機械検査を RBAC/HTTP 型の混入まで拡張 | §2 | S | **不採用** — §4.1 |
| CR-5 | §6 の「レビュー担保」項目のうちテスト固定できるものをテスト化 | §6 | M | **機会的** — §4.1 |

**機械検査はここで打ち止め（CR-1 / CR-2 で本丸を塞いだ）。** 理由と CR-3〜5 の
個別判断は次節。

### 4.1 打ち止めの判断（2026-07-21）

機械検査には実装コストだけでなく**継続コスト**がある — 誤検知しうる / 将来の
リファクタを縛る / 保守が要る / 新しい担当（人でもAIでも）が理解する必要がある。
したがって機械化する価値がある不変条件は、次の3条件を**同時に**満たすものに
絞る:

1. **背骨である**（壊れると重大）
2. **壊れても静かに壊れる**（レビューやビルドで気づけない）
3. **AIが無自覚に壊しうる**

CR-1（両経路対称）はこの3つを完全に満たす（＝明確な当たり）。CR-2 は半分
（settings 対称は妥当、mime も安価なので許容）。**実質的な収穫はここまでで
取り切っている。** これ以上の機械検査追加はガードレール自体が保守負担になり、
偽の安心感を生む側に傾く。本リポジトリの「実需が出た時点で着手」方針
（[improvement-plan-2026-07.md](improvement-plan-2026-07.md) P4-5 等）とも整合する。

個別判断:

- **CR-4（§2 を RBAC/HTTP 型混入まで拡張）— 不採用。** サービス層は小さく
  `use axum`/`use tauri` で既に守られている（rule 1）。追加で捕まえる退行の確率が
  低く、「何を HTTP 型とみなすか」の線引きで誤検知が出やすい。ノイズを増やす
  だけで条件2・3を満たさない。
- **CR-5（§6 レビュー項目のテスト固定）— 機会的にのみ。** テスト自体は善だが、
  対象（body-limit 順序・security headers LAST・監査 detail に秘密を入れない）は
  意味のあるテストが書きにくく、一部は暗黙にカバー済み。労力対効果が薄い。
  **当該コードを触るとき**にそのついでで固定するに留める（新規タスクにしない）。
- **CR-3（依存追加の検出）— 実需ドリブンで見送り。** 「依存を足さない」は文化の
  中核（ADR-0002 / §3）で、AI はライブラリに手を伸ばしがちなので条件3は満たす。
  一方、依存追加は `Cargo.toml`/`package.json` の**目に見える意図的な差分**であり、
  grid-core のような静かなドリフトとは違う（条件2 が弱い）。許可リスト方式は正規の
  追加のたびに更新が要り摩擦が形骸化を招く。**実際に依存が増え始めた/その傾向が
  見えた時点**で、軽量なスナップショット差分方式で着手する（それまで着手しない）。

要するに: **CR-1・CR-2 = 実施済み。CR-3 = 実需が出たら軽量版。CR-4 = やらない。
CR-5 = コードを触るついでに。** 打ち止めを覆すのは「実際に該当の退行や実需が
観測されたとき」に限る。

### CR-1 の設計方針（実装前に実現性を1枚確認する）

- **狙い**: 「新しい mutating 操作を片方の経路にだけ足す」ミスを CI で検出。
- **抽出**: Tauri 側は `src-tauri/src/lib.rs` の `#[tauri::command]` 関数名、
  REST 側は `rest/*.rs` のルート定義（`.route(...)` のメソッド + パス）から
  mutating 集合を機械抽出する。読み取り系（list/get）は conventions §1 に従い
  対称の対象だが監査対象外なので、mutating（create/update/delete/import/
  login/logout/setup 等）に絞るか、全対応を見るかは実現性確認で決める。
- **照合**: 両集合を正規化した名前で突き合わせ、片側にしか無いものを違反として
  報告。命名の対応（`items_create` ⇔ `POST /api/items`）は誤検知しやすいため、
  **対応表を明示（許可リスト/マッピング）する方式**を第一候補とし、完全自動
  推論はしない。
- **監査照合（任意・第2段）**: 各 mutating コマンド/ハンドラに監査記録
  （`.record(AuditEntry{..})` / `record_write`）呼び出しがあるかを併せて検査。
- **誤検知対策**: `src-tauri` はサンドボックスでコンパイルできないため、検査は
  **静的テキスト解析**（正規表現）に留める。曖昧な対応は許可リストで明示し、
  「検査が通ること」より「片側追加を確実に捕捉すること」を優先する。
- **受け入れ条件**: 現状の両経路（47 Tauri コマンド ⇔ 対応する REST ルート）に
  対して緑。ネガティブテスト（片側にダミー mutating を足す）で赤になること。

### CR-1 実装結果（2026-07-21、rule 8）

上記方針どおり実装した。

- **一次情報のアンカー**: (1) `src-tauri/src/lib.rs` のトップレベル
  `#[tauri::command] fn`（47件。tests は属性を持たず混入しない）、(2)
  `rest/mod.rs` モジュール doc の Route table（36ルート。§1 が対応表と指定する
  成果物なのでこれを機械検査対象に昇格）。どちらも test ノイズ無く確実に
  パースできる。
- **構成**: `DUAL_PATH` マニフェスト（20対、所有者確認済みの分類 §3）+ 完全性
  チェック。実 `.route()` 宣言（app rest/ + `banto-server`）との doc-sync も併設。
- **捕捉できる失敗（4種すべてネガティブテストで赤を確認）**: (a) dual-path の
  片側が消える/改名、(b) 未分類の Tauri コマンド追加、(c) 未分類の REST ルート
  追加、(d) Route table と実ルートのドリフト。→ 「片側だけに mutating を足す」は
  (b)(c) で必ず捕捉。
- **捕捉しないもの**: 「両経路に存在するか」までで、**認可レベル・監査記録が
  実際に同一か**（同じロール床・同じ `AuditEntry`）はコード + テスト担保
  （CR-5 でテスト固定を検討）。
- **分類を変えるとき**: desktop-only / read の判断は §3 が根拠。新 mutating 操作を
  足すときは `DUAL_PATH` に1行足すだけ（両端の実在を CI が検証）。

## 5. 非スコープ（今回は記録のみ）

- 大きいファイル（`auth.rs` 1420 / `backup.rs` 1027 / `users.rs` 1017）の分割 —
  凝集はしており（1ドメイン1ファイル）今すぐ不要。「なぜ」コメントの密度を
  §2 モデル水準に揃える方が価値が高い。
- フロントパッケージ群・アプリ配線の保守性レビュー — 別途スコープ。

## 6. まとめ

- 本レビューは AI 中心保守を前提に、**機械強制されていない不変条件**に絞った。
- 最優先は §1 両経路対称の機械検査（CR-1）。AI の最大の盲点を塞いだ。
  §6 の grep 可能な2件（CR-2）も機械化した。
- 方向性は rule 7 と同じ「散文の規約 → 落ちる検査」。依存を足さない。
- **CR-1 / CR-2 で本丸は塞いだので、機械検査の追加はここで打ち止め**（§4.1）。
  ガードレール自体が保守負担・偽の安心感になる手前で止める。残る CR-3〜5 は
  実需/機会ドリブンのバックログとして残し、実際の退行や実需が観測された時点で
  だけ着手する。コードベースは元々良い状態で、レビューの実質収穫は CR-1 と
  （前段の）ドキュメントドリフト修正だった、という点も正直に記録する。
